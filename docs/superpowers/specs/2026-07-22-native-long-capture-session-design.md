# 原生长截图会话与边框窗口设计

## 背景与实机证据

当前实现让全屏截图 WebView 在长截图期间继续存在，并同时承担选区边框展示、鼠标穿透、截图排除和取消清理。实机检查得到以下证据：

- 长截图期间，选区中心的 Win32 命中测试可以落到下层应用，模拟滚轮也能推动下层 Codex 页面；因此问题不只是前端 `pointer-events`。
- 全屏截图窗口和长截图侧栏都被设置为 `WDA_EXCLUDEFROMCAPTURE`，但退出长截图时没有对称恢复。窗口状态会跨截图会话残留，录屏与部分捕获路径可得到纯黑内容。
- Esc 由全局快捷键、前端键盘事件和异步长截图任务分别处理。窗口隐藏、React 会话重置和运行时结束没有一个统一的完成边界，存在先后顺序竞争。
- 自动化复现中 Esc 后再次截图能够得到空选区，说明残留选区是时序相关问题，不能再用只覆盖理想顺序的前端测试作为充分证明。

## 目标

- 长截图期间，所选区域内不存在任何截图工具全屏窗口，滚轮及其他输入直接交给原应用。
- 保留与微信 4.1.11 相近的绿色选区边界，但边界窗口不能覆盖选区内部。
- Esc、取消按钮和异常退出走同一个原生幂等清理流程。
- 取消完成后再允许新截图；下一次截图必须从空选区开始。
- 所有临时窗口的捕获排除标记都随窗口生命周期清除；主截图窗口恢复为 `WDA_NONE`。
- 普通截图的选区外遮罩只有一层，黑色 alpha 固定为 `0.3`，选区内不叠加遮罩。

## 非目标

- 不修改手动滚动识别、重叠匹配和拼接阈值。
- 不恢复自动滚动。
- 不修改 OCR、翻译、隐私打码、标注工具或工具栏图标。
- 不增加蒙层透明度设置项。

## 方案比较

### 方案 A：继续保留全屏透明 WebView

改动最少，但仍依赖 WebView 透明合成、原生鼠标穿透、捕获排除和焦点状态同时正确。当前三类故障正发生在这些状态的组合处，因此不采用。

### 方案 B：长截图期间完全隐藏选区边界

只保留侧栏，输入路径最简单，但与已确认的微信式交互不一致，用户也无法确认当前捕获范围，因此不采用。

### 方案 C：隐藏全屏蒙层，创建四条窄型边框窗口

长截图期间隐藏全屏 WebView，只在选区的上、右、下、左边缘创建四个不可激活、鼠标穿透、排除捕获的窄窗口；侧栏继续位于选区外。该方案既保留视觉边界，也从结构上保证选区内部没有截图工具窗口。用户已批准采用此方案。

## 窗口架构

### 普通截图阶段

- `overlay` 是唯一全屏截图窗口。
- 每次开始截图时，原生层先发布新的会话编号并要求前端重置，再隐藏窗口采集桌面帧。
- 桌面帧准备好后才显示并聚焦 `overlay`。
- `overlay` 必须处于可捕获默认状态 `WDA_NONE`；普通截图不依赖捕获排除。

### 长截图阶段

- 在隐藏 `overlay` 前，将逻辑选区换算成物理屏幕坐标并定位下层目标窗口。
- 创建 `scroll-capture-preview` 侧栏，布局保持在选区外侧。
- 创建四个边框窗口：`scroll-border-top`、`scroll-border-right`、`scroll-border-bottom`、`scroll-border-left`。
- 边框厚度为 1 个物理像素；可在高 DPI 下按不低于 1 物理像素处理，但不得向选区内部扩张。
- 四个边框窗口和侧栏均设置为置顶、无装饰、不出现在任务栏、不可激活。边框窗口启用鼠标穿透。
- 临时窗口创建成功并完成捕获排除后，隐藏 `overlay`，等待 150 ms，再采集第一帧。
- 若任一临时窗口准备失败，关闭全部已创建临时窗口、恢复 `overlay`，且不采集第一帧。

### 为什么滚轮一定能穿透

选区内部不再存在全屏透明窗口或大型预览窗口。四条边框仅占据选区外沿的窄矩形，侧栏也与选区矩形不相交。因此选区内的 Win32 命中测试无需依赖 `WS_EX_TRANSPARENT` 的全屏窗口行为，天然指向被截图应用。

## 原生会话状态

新增一个由原生层拥有的长截图窗口会话，至少包含：

- 是否正在准备或捕获；
- 当前会话编号；
- 停止与取消标记；
- 已创建的侧栏和边框窗口标签；
- 清理是否已经开始或完成。

状态流为：

`Idle -> Preparing -> Capturing -> Finishing -> Idle`

任何准备失败、Esc 或取消请求都进入 `Finishing`。`Finishing` 期间拒绝新的截图快捷键，直到清理完成并回到 `Idle`。

## 取消与清理顺序

所有取消入口只调用一次原生 `cancel` 操作。原生层按以下顺序幂等清理：

1. 设置取消标记，使采集循环停止且丢弃结果。
2. 关闭侧栏和四个边框窗口。
3. 将 `overlay` 的捕获排除恢复为 `WDA_NONE`，并关闭鼠标穿透。
4. 向 `overlay` 定向发送带会话编号的 `capture-session-reset`，前端清空选区、标注历史、长截图资源和错误状态。
5. 等待重置事件已经排入目标窗口事件队列后隐藏 `overlay`。
6. 注销仅用于长截图的 Escape 快捷键，标记会话清理完成。

重复到达的 Esc、前端取消和侧栏取消只观察同一个清理结果，不重复关闭窗口，也不把状态恢复为旧选区。

普通的编辑、保存和完成保留拼接结果，但仍通过同一资源清理器关闭临时窗口、恢复 `overlay` 的系统状态，然后进入各自输出流程。

## 新截图会话隔离

- 原生截图请求先检查长截图会话是否为 `Idle`；清理期间忽略快捷键。
- 每次截图开始生成单调递增的会话编号。
- `capture-started`、`capture-ready` 和 `capture-session-reset` 都携带会话编号。
- 前端只接收当前或更新的会话事件，迟到的旧长截图结果和旧重置事件不能覆盖新会话。
- `ScreenshotEditor` 仍以会话编号作为 React `key`，因此新截图在组件层重新挂载并从 `selection: null` 开始。

## 蒙层渲染

- 无选区时，使用一个覆盖全屏的 `rgba(0, 0, 0, 0.3)` 遮罩。
- 有选区时，不再同时依赖全屏背景和超大阴影。改用四个独立遮罩矩形覆盖选区的上、右、下、左区域，每块均为 `rgba(0, 0, 0, 0.3)`，矩形之间不重叠。
- 选区内部完全不渲染遮罩。
- 长截图期间 `overlay` 被隐藏，不再通过 CSS 透明类模拟消失。

## 组件边界

- `platform/windows/capture_exclusion.rs`：提供设置 `WDA_EXCLUDEFROMCAPTURE` 与恢复 `WDA_NONE` 的对称接口。
- `preview_windows.rs`：计算侧栏和四条边框的物理矩形，创建临时窗口。
- `long_capture.rs`：拥有状态机、准备顺序、采集循环和幂等清理。
- `app_state.rs`：在长截图清理期间拒绝普通截图请求，并发布带编号的截图事件。
- `App.tsx`：按会话编号过滤事件并重新挂载编辑器。
- `SelectionOverlay.tsx`：用四块不重叠遮罩实现精确 `0.3` alpha。

## 错误处理

- 任何窗口创建、捕获排除或坐标转换失败都在第一帧前中止，并恢复普通截图窗口。
- 恢复捕获标记失败时继续关闭其他资源，同时记录并显示具体 Windows 错误。
- 取消不显示“长截图失败”；真正的准备或拼接错误恢复编辑器并显示错误。
- 清理中的单步失败不能阻断后续步骤，最终必须释放运行时活动标记。

## 测试策略

### 自动化测试

- Rust：四条边框矩形不进入选区内部，并位于正确的物理屏幕坐标。
- Rust：准备顺序为定位目标、创建临时窗口、排除捕获、隐藏主窗口、首帧。
- Rust：取消清理顺序固定且重复取消幂等。
- Rust：捕获排除接口使用 `0x11`，恢复接口使用 `0x0`。
- Rust：长截图为非空闲状态时普通截图请求被拒绝。
- React：旧会话事件不能恢复选区；新 `capture-started` 立即得到空选区。
- React/CSS：四块遮罩不重叠，颜色均为 `rgba(0, 0, 0, 0.3)`，不再使用 `9999px` 阴影。

### Windows 实机验收

1. 启动工具并在可滚动页面选择区域。
2. 进入长截图，确认全屏 `overlay` 已隐藏，只有四条绿色边框和选区外侧栏可见。
3. 在选区中心验证 `WindowFromPoint` 属于被截图应用；连续滚轮能推动页面并增加拼接高度。
4. 按 Esc，确认所有临时窗口一次消失，`overlay` 隐藏，显示亲和性恢复为 `0x0`。
5. 立即再次使用 `Alt+Shift+A`，确认必须重新拖选，且不存在旧工具栏或选区。
6. 普通截图无选区及有选区两种状态下取样像素，验证选区外为单层 0.3 黑色、选区内与源图一致。
7. 重复上述流程至少三次，排除偶发时序问题。

## 验收标准

- 用户报告的三个问题连续三轮不可复现。
- 长截图期间选区内部没有属于截图工具进程的顶层或子窗口命中。
- 退出后 `overlay` 的显示亲和性为 `WDA_NONE`，无残留侧栏或边框窗口。
- 新截图会话的初始选区恒为 `null`。
- 前端测试、Rust 测试、类型检查、格式检查、Clippy、生产构建和 Tauri debug 构建全部通过。
